# SOP From Master — 从主R操作生成标准化流程

## 设计哲学

来自美团实践：**"主R先跑通最复杂包 → 沉淀可 AI 执行的 SOP → 全组并行"**。

将主R的一次成功重构/迁移转化为可复用的操作模板，其他成员拿着 SOP 指导 AI 完成类似模块改造。效果：**一个人的上限 → 全组的基线**。

## 核心原则

1. **先跑通再沉淀**：SOP 必须基于真实成功操作记录，不凭空设计
2. **模板化**：文件路径、类名替换为 `{Entity}` 等模板变量
3. **每个步骤必须有验收标准**：如何判断这一步做对了
4. **SOP 是消耗品**：随代码演进而过期，不需要长期维护

## 执行流程

### Phase 1: 分析主R的操作记录

```
1. git log --oneline --author="主R" --since/--until 获取 commit 序列
2. 对每个 commit: git diff --stat 获取变更文件和类型
```

AI 将变更归类为操作类型：

| 操作类型 | 识别特征 |
|---------|---------|
| **提取类** | 新建文件 + 从其他文件移入方法 |
| **重命名** | 仅名称变化，body 不变 |
| **拆分方法** | 大方法变短 + 新增多个小方法 |
| **移动文件** | git mv 或内容不变路径变 |
| **修改签名** | 方法参数/返回值类型变化 |
| **添加接口** | 新增 interface/abstract class |
| **修改依赖** | import 语句变化 |

### Phase 2: 抽象为可 AI 执行的 SOP

#### 抽取模板变量

| 具体值 | 模板变量 |
|--------|---------|
| UserService.java | `{Entity}Service.java` |
| UserConverter.java | `{Entity}Converter.java` |
| com.company.order | `{BasePackage}.{Module}` |

#### 生成 SOP 文档

```markdown
# SOP: {任务名称}
> 生成自: 主R [{名字}] 在 [{日期}] 对 [{模块}] 的操作
> 适用模块: {适用范围}
> 预计每个模块耗时: {预估时间}

## 前置条件
- [ ] 已阅读项目的 CLAUDE.md 架构约束
- [ ] 当前模块全量测试通过（基线）

## 操作步骤

### Step N: {步骤名称}
- 目标文件: {模板化路径}
- 操作类型: {提取类/重命名/拆分/...}
- 操作规则: {具体的代码转换规则}
- 验收标准: {如何验证做对了}

## 常见问题
| 问题 | 根因 | 解法 |
[主R实际操作中遇到的问题]

## 完成检查清单
- [ ] 所有步骤验收标准通过
- [ ] 全量测试通过
- [ ] Pre-PR Check 无 CRITICAL/HIGH
```

### Phase 3: 团队验证

1. 另一成员在新模块上按 SOP 执行
2. 记录遇到的问题和偏差
3. 迭代更新 SOP

### Phase 4: SOP 生命周期

- **位置**: `.claude/sops/SOP-{序号}-{简述}.md`
- **过期标记**: 标注适用版本范围和过期条件
- **归档**: 过期后移到 `.claude/sops/archive/`

## AI 执行 SOP 行为准则

1. **逐步骤执行**：Step 1 验收通过后才能进入 Step 2
2. **遇到偏差立即停止**：实际情况与 SOP 不一致时，报告差异，由人决定如何继续
3. **不跳过验收**：每个步骤的验收标准必须全部通过
4. **记录执行日志**：每个步骤完成后记录成功/失败/偏差
