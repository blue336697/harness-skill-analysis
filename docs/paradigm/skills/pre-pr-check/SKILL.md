# Pre-PR Check — 提交前 AI 自查

## 目的

在代码进入人工 CR 之前，AI 先行对变更代码做 7 个维度的穷举检查，过滤掉所有"机器能发现的低级问题"。确保 Reviewer 拿到的代码只需要关注"业务语义是否正确"和"架构决策是否合理"。

## 核心原则

1. **穷举，不抽样**：对所有变更文件的所有行做检查，不遗漏
2. **只报告，不修改**：输出问题清单，由人决定哪些需要修复
3. **分级明确**：CRITICAL（阻塞）→ HIGH（应修复）→ MEDIUM（建议）→ LOW（可选）
4. **生成 PR 文档**：最终输出标准 PR 描述模板

## 执行流程

### Phase 1: 变更范围扫描

```
1. 确定 base 分支（默认 origin/main 或用户指定）
2. git diff --name-only origin/main...HEAD 列出所有变更文件
3. 对每个变更文件获取具体变更行范围
```

输出变更清单：文件数、新增/修改/删除分类。

### Phase 2: 七维度自查

对每个变更文件逐维度检查：

#### 维度 1: 规范合规
- 新代码是否符合项目 CLAUDE.md 的分层架构约束？（检查 import 方向）
- 命名是否符合规范？（PascalCase/camelCase/UPPER_SNAKE_CASE，无拼音）
- 是否有硬编码的魔法数字/字符串？（排除 0, 1, -1）
- 文件是否超过行数限制？

#### 维度 2: Bug 扫描
- 所有错误路径是否都有处理？（搜索空 catch 块）
- null/undefined 检查是否完整？
- 边界条件是否覆盖？（空集合、除零、索引越界）
- 并发安全：共享可变状态是否有同步保护？

#### 维度 3: 异常处理
- 是否有空的 catch 块？→ CRITICAL
- 重新抛出的异常是否保留了原始堆栈(cause)？
- 错误信息是否包含调试上下文（入参、状态码）？
- 外部调用（API/DB/Cache）是否有 try-catch？

#### 维度 4: 一致性
- 新代码模式是否与同模块已有代码一致？（返回值类型、事务标注、日志格式）
- 是否引入了与现有模式冲突的新模式？

#### 维度 5: 可扩展性
- 是否存在 flag 参数切换逻辑的反模式？
- 是否有 >5 分支的 if-else/switch？（考虑策略模式）
- 是否存在"上帝类"？

#### 维度 6: 性能
- 是否存在 N+1 查询？（循环内 DB/RPC 调用）
- 是否缺少分页？
- 字符串拼接是否优化？

#### 维度 7: 死代码与残留清理 ★ 重构必查
- **新旧并存**：新增了功能等价的类/方法，但旧代码未删除？
  - 检查方式：对比新增文件和修改文件，如果新增了 `NewOrderService`，检查是否存在 `OldOrderService` 仍被保留且功能重叠
- **删除不完整**：删除了某个类/方法，但它的 import、配置、测试引用仍残留？
  - 检查方式：codegraph_callers 确认删除的符号是否还有调用方
- **注释掉的代码**：是否有连续 3+ 行的注释代码块？（应该用 git history 追溯，不是注释保留）
- **废弃文件**：是否存在内容为空、或只剩 import 但无实际逻辑的文件？
- **孤立的 import**：import 的类/模块已被删除但 import 语句残留？
- **遗留的旧配置**：旧框架/旧模块的配置文件、注解、XML 映射是否一并清理？

**分级规则**：
- 新旧功能等价代码并存 → CRITICAL（阻塞合并，必须清理）
- 删除不完整（残留 import/配置引用）→ HIGH（编译可能通过但留隐患）
- 注释掉的代码块 → MEDIUM
- 废弃的空文件 → MEDIUM

### Phase 3: 生成自查报告

按严重级别输出，每项包含：文件:行号、维度、描述。

### Phase 4: 修复循环

1. 展示报告，询问是否修复 CRITICAL + HIGH
2. 逐项修复 → 验证 → 下一项
3. MEDIUM/LOW 保留在 PR 描述中供 Reviewer 参考

### Phase 5: 生成 PR 文档

自动生成标准 PR 描述：
- 改动概述
- 变更文件表（路径、类型、说明）
- 影响范围
- 需重点 Review 的业务逻辑
- Pre-PR Check 结果统计
- 未修复的 MEDIUM 问题及原因
- 测试情况
